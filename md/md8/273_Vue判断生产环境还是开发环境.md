# Vue 中判断当前是生产环境还是开发环境？

## Vue 2

在 Vue 2 中，可以通过检查 `process.env.NODE_ENV` 变量来确定当前是生产环境还是开发环境。在开发环境下，`process.env.NODE_ENV` 的值通常是 `"development"`，而在生产环境下，它通常被设置为 `"production"`。

```
if (process.env.NODE_ENV === 'production') {
  // 生产环境逻辑
  console.log('这是生产环境');
} else {
  // 开发环境逻辑
  console.log('这是开发环境');
}
```

请注意，在生产环境下，许多构建工具（比如 Webpack）会自动将 `process.env.NODE_ENV` 设置为 `"production"`。而在开发过程中，通常不需要手动设置此变量，它会被构建工具自动识别。

## Vue 3 + Vite

在 Vue 3 + Vite 中，可以使用 Vite 提供的 `import.meta.env` 对象来判断当前是生产环境还是开发环境。这是 Vite 特有的一个对象，用于访问环境变量。

```
if (import.meta.env.MODE === 'production') {
  // 生产环境逻辑
  console.log('这是生产环境');
} else {
  // 开发环境逻辑
  console.log('这是开发环境');
}
```

在上面的代码中，`import.meta.env.MODE` 可以用来获取当前的模式，其值为 `'development'` 或 `'production'`。这里的 `'development'` 对应开发环境，而 `'production'` 对应生产环境。

确保在你的 Vite 项目中使用这个方式来判断环境。这是 Vite 特定的方法，而在其他构建工具（如 Vue CLI）中可能有不同的方式。

### 注意事项

* **构建时替换**：`process.env.NODE_ENV` 和 `import.meta.env` 的值是在构建时静态替换的，因此不要在运行时动态修改它们。



## 构建时替换

**这些变量并不存在于浏览器的运行时环境中，而是在代码打包阶段，被构建工具（Webpack/Vite）直接“原地替换”成了具体的字符串字面量（如 'production' 或 true）。**

为了让你看得更透彻，我从底层机制、代码转换过程、以及常见误区三个层面展开：

### 1. 底层机制：构建工具的“查找与替换”

本质上，这是一个**文本级别的 AST（抽象语法树）节点替换**过程。

- **Webpack（Vue 2 / Vue CLI）**：使用 **DefinePlugin** 插件。它的工作原理是，在编译 JavaScript 时，遇到 `process.env.NODE_ENV` 这个标识符，就直接把它替换成对应的字符串字面量。
- **Vite（Vue 3）**：底层使用 **esbuild** 进行预构建，Rollup 进行打包。它们提供了 `define` 配置项，执行完全相同的静态替换逻辑。

------

### 2. 直观的代码“变形记”

为了让你看清“替换”长什么样，我们来看一个实际发生的代码转换过程：

**你编写的源代码（输入）：**

```javascript
if (process.env.NODE_ENV === 'production') {
  console.log('调用真实API');
  fetch('/api/pay');
} else {
  console.log('调用Mock数据');
  fetch('/mock/pay');
}
```

**当你执行 npm run build 构建生产包时，构建工具会进行文本替换。由于生产环境下 NODE_ENV 被设置为 "production"，上面的代码会被“硬替换”为：**

```javascript
if ('production' === 'production') {  // 注意：process.env.NODE_ENV 消失了！
  console.log('调用真实API');
  fetch('/api/pay');
} else {
  console.log('调用Mock数据');
  fetch('/mock/pay');
}
```

**紧接着，构建工具会进行“死代码消除（Tree Shaking / Dead Code Elimination）”：**

因为 `'production' === 'production'` 永远是 `true`，`else` 分支永远不会执行，所以构建工具会认为 `else` 里面的代码是“死代码”，直接将其**彻底删除**。

**最终打包出来的生产代码（输出）实际只剩下了：**

```javascript
console.log('调用真实API');
fetch('/api/pay');
```

你看，不仅变量没了，连开发环境的逻辑代码都被物理删除了。这就是“静态替换”带来的最大收益——**零运行时性能开销**。

------

### 3. 必须警惕的“静态”陷阱（重要！）

既然是“静态文本替换”，就意味着**它不认识 JavaScript 的变量和对象动态属性**。这是初学者最容易犯错的地方：

- ❌ **错误写法（动态取属性）**：

  ```javascript
  const env = process.env;
  if (env.NODE_ENV === 'production') { ... }
  ```

  因为构建工具只匹配完整的 `process.env.NODE_ENV` 字符串，写成 `env.NODE_ENV` 后，构建工具找不到匹配项，**替换会失效**。最终打包时 `env.NODE_ENV` 在浏览器中运行会直接报错（因为浏览器没有 `process` 对象）。

- ❌ **错误写法（解构赋值）**：

  ```javascript
  const { NODE_ENV } = process.env;
  if (NODE_ENV === 'production') { ... }
  ```

  同样会失效，原因同上。

- ✅ **正确写法（必须写全名）**：

  ```javascript
  if (process.env.NODE_ENV === 'production') { ... }
  // Vite 中
  if (import.meta.env.PROD) { ... }  // 这是 Vite 封装好的布尔值，同样会被替换为 true/false
  ```

------

### 4. 特别澄清：Vite 中的 `import.meta.env`

在 Vite 中，`import.meta` 本身是 ES Module 的原生语法。但 `import.meta.env` 并不是浏览器内置的，它同样是 Vite 在构建时**注入**的。

- 当你写 `if (import.meta.env.MODE === 'production')`，Vite 会把这段代码替换为 `if ('production' === 'production')`。
- Vite 还贴心地提供了 `import.meta.env.DEV` 和 `import.meta.env.PROD`，它们在构建时直接被替换为字面量 `true` 或 `false`，所以你甚至可以写 `if (import.meta.env.PROD)`，最终 `if(true)` 或 `if(false)` 会被优化掉。

------

### 5. 与真正的“运行时变量”区分

为了让你更清晰地理解，补充一个对比：

| 类型           | 机制                       | 生效时机              | 典型场景                                                    |
| :------------- | :------------------------- | :-------------------- | :---------------------------------------------------------- |
| **构建时替换** | 打包工具直接修改代码文本   | 代码打包阶段          | 区分开发/生产接口地址、删除调试日志（`console.log`）        |
| **运行时变量** | 从服务器或浏览器环境中读取 | 页面加载后（Runtime） | Docker 镜像部署时配置数据库密码、Nginx 反向代理注入的请求头 |

如果你需要在**部署后**（不重新打包）动态切换接口地址，就不能用 `process.env`，而需要在 `index.html` 中通过 `<script>` 挂载全局变量（如 `window.GLOBAL_CONFIG`），或者通过后端接口获取配置。



## npm run build

`npm run build` 就像是一个“启动按钮”，真正干活的其实是 `package.json` 中 `scripts` 字段里定义的命令（比如 `vue-cli-service build` 或 `vite build`）。

为了让你看清黑盒里到底发生了什么，我把整个过程拆解为 **“准备 → 编译 → 优化 → 产出”** 四个阶段，并带入 Webpack（Vue2/CLI）和 Vite（Rollup）的视角。

------

### 阶段零：触发与配置解析（CLI 启动）

当你敲下回车，终端会执行以下操作：

1. **查找命令**：npm 在当前项目的 `node_modules/.bin/` 中找到对应的构建工具（如 `vue-cli-service` 或 `vite`）。
2. **加载配置**：读取项目根目录的配置文件（`vue.config.js`、`vite.config.js` 或 `webpack.config.js`），并合并 **默认配置**。
3. **设置环境变量**：**重点来了**——构建工具会强行将 `process.env.NODE_ENV` 设置为 `"production"`（这就是之前我们聊的“构建时静态替换”的数据来源）。

------

### 阶段一：构建依赖图谱（解析入口）

构建工具会从**入口文件**（Entry）开始，通常是 `src/main.js`（Vue）或 `src/index.js`（React）。

- **AST 解析**：工具不会逐行读文本，而是将代码解析成 **AST（抽象语法树）**，精准识别所有的 `import` 和 `require`。
- **递归追踪**：从入口出发，像蜘蛛网一样把项目中所有关联的 `.js`、`.vue`、`.ts`、`.css`、图片等文件全部找出来，生成一张**依赖图谱（Module Graph）**。

------

### 阶段二：模块转换（Loader / Plugin 处理）

原始代码浏览器不认识（比如 Vue 组件、TS、SCSS），必须在这里进行“翻译”：

- **Vue 组件**：`vue-loader` 将 `.vue` 文件拆解成 `template`（编译为 render 函数）、`script` 和 `style`。
- **JS/TS 转译**：`babel-loader` 或 `esbuild` 将 ES6+ / TypeScript 语法降级为浏览器兼容的 ES5/ES6 代码。
- **CSS 处理**：`css-loader` + `style-loader`（开发）或 `MiniCssExtractPlugin`（生产）提取样式。
- **静态资源**：图片和字体小于一定体积（如 4kb）时，被转为 **Base64** 内联；大的则生成独立文件并加哈希值。

------

### 阶段三：代码优化（生产环境专属，最耗时的一步）

这是生产构建**最核心**的环节，工具会“不择手段”地让代码变得最小、最快。**Vite 使用 Rollup 插件，Webpack 使用内置优化项**：

| 优化动作                       | 具体做了什么                                                 |
| :----------------------------- | :----------------------------------------------------------- |
| **Tree Shaking（摇树）**       | 利用 ES Module 静态结构，删除项目中**未被引用**的“死代码”。比如 `lodash` 中只用了 `cloneDeep`，其他函数会被直接丢弃。 |
| **代码压缩（Minify）**         | Webpack 用 `TerserWebpackPlugin`，Vite 用 `esbuild` 或 `terser`。去掉空格、缩短变量名（如 `function getUser` 变成 `function a`），并将 `true && foo` 简化为 `foo`。 |
| **静态替换**                   | 将 `process.env.NODE_ENV` 替换为 `"production"`，并执行**死代码消除**。比如 `if(process.env.NODE_ENV === 'development')` 里面的 `console.log` 会被整段删除。 |
| **代码分割（Code Splitting）** | 将依赖的第三方库（如 `vue`、`element-plus`）单独打包成 `vendor.[hash].js`，业务代码打成 `app.[hash].js`。这样浏览器可以利用缓存，库没变就无需重新下载。 |
| **CSS 提取与压缩**             | 将所有 CSS 抽离成独立的 `.css` 文件（而不是塞进 JS 里），并用 `cssnano` 压缩样式。 |

------

### 阶段四：文件输出与指纹标识（Hash）

优化完成后，构建工具将最终内容写入磁盘（默认 `dist` 文件夹）：

1. **命名哈希（缓存破坏）**：文件名会带上 `contenthash`（如 `app.a3f5c2.js`）。只要文件内容变一次，哈希就变一次，完美解决浏览器缓存问题。
2. **HTML 注入**：`HtmlWebpackPlugin` 或 `vite-plugin-html` 会自动把生成的 JS/CSS 文件路径，以 `<script>` 和 `<link>` 标签的形式注入到 `index.html` 中。
3. **静态资源拷贝**：`public` 目录下不需要编译的静态文件，被原封不动复制到 `dist` 根目录。

------

### 🔍 Vue CLI 和 Vite 的特殊差异

虽然流程相似，但底层机制决定了它们的大致耗时：

| 维度         | Vue CLI (Webpack)                 | Vite (Rollup)                                                |
| :----------- | :-------------------------------- | :----------------------------------------------------------- |
| **核心思想** | 将所有代码打包成一个或多个 Bundle | 利用原生 ES Module，打包时主要做**混淆压缩**和**代码拆分**   |
| **构建速度** | 相对较慢（所有代码都要重新编译）  | 极快（底层用 Go 写的 esbuild 做预构建，Rollup 只做最终打包） |
| **产物格式** | 默认 ES5（兼容性好，代码较臃肿）  | 默认现代 ES Module（更精简，但需现代浏览器）                 |

------

### 最终产物（Dist 文件夹里有什么？）

执行完毕后，你的 `dist` 文件夹大致是这个结构：

```text
dist/
├── index.html        （入口，引用了下面所有带 hash 的文件）
├── css/
│   └── app.7f8a9b.css
├── js/
│   ├── vendor.8h3k2.js   （第三方库）
│   ├── app.a3f5c2.js     （业务代码）
│   └── chunk-xx.9d4e.js  （按需加载的路由/组件）
└── img/
    └── logo.2e3f.svg
```



## npm run dev

如果说 `npm run build` 是一场“精心准备的正式演出（追求极致性能）”，那 `npm run dev` 就是一场“即兴排练（追求极致开发效率）”。

它的核心目标不是产出文件，而是**启动一个开发服务器，提供实时调试环境**。具体过程如下：

------

### 阶段零：命令触发与环境设置

和 Build 一样，`npm run dev` 会调用 `node_modules/.bin/` 下的对应命令（如 `vue-cli-service serve` 或 `vite`）。

- **核心差异**：构建工具会**强行将 process.env.NODE_ENV 设置为 "development"**（而不是生产模式）。
- **读取配置**：加载 `vue.config.js` 或 `vite.config.js`，但会合并开发专用配置（如更严格的 Source Map、更少的缓存策略）。

------

### 阶段一：启动开发服务器（核心分水岭）

从这一步开始，**Webpack（Vue CLI）** 和 **Vite（Vue 3）** 的底层逻辑出现了巨大分歧：

- **Webpack（Vue CLI）—— 全量打包**：
  它会像 Build 一样，从入口文件开始，**递归构建完整的依赖图谱**，将所有的 `.js`、`.vue`、`.css` 全部打包成 `bundle.js`。只不过：
  1. 打包结果不写入磁盘，而是存放在**内存（Memory FS）**中。
  2. 打包完成后，才会启动 `webpack-dev-server` 监听端口。
     *（代价是：项目越大，启动越慢，需要等待全部编译完）*
- **Vite —— 按需加载（原生 ESM）**：
  Vite 利用了现代浏览器支持 `<script type="module">` 的特性，**不会立即打包所有代码**。
  1. 它启动一个极快的 **HTTP 服务器**。
  2. **预构建（Pre-bundling）**：使用 Go 语言写的 `esbuild` 快速将第三方依赖（如 `vue`、`element-plus`）转换为单文件，缓存起来。
  3. **不打包业务代码**：源文件的 `import` 语句被保留，浏览器请求哪个文件，Vite 就实时编译哪个文件（极速启动）。

------

### 阶段二：按需编译与转换（No Bundle / Bundle）

当浏览器访问 `http://localhost:8080` 请求 `main.js` 时，构建工具开始“上课回答问题”：

- **模块转换**：`.vue` 文件被 `vue-loader`（Webpack）或 `@vitejs/plugin-vue`（Vite）解析成 JS 对象；`.ts` 被转译为 JS；`.scss` 被转为 CSS。
- **关键特征 —— 跳过优化**：
  - **不做 Tree Shaking**（保留所有未引用的代码，因为开发时随时可能改动并热更新）。
  - **不做代码压缩（Minify）**：保留完整的变量名（如 `getUserInfo` 不会被简化为 `a`），方便报错时查看堆栈。
  - **不合并小图片**：图片不会被转成 Base64 内联，保持原样请求，减少编译压力。

------

### 阶段三：热模块替换（HMR）—— 开发体验的灵魂

这是 Build 阶段**完全没有**的环节。当你修改了 `.vue` 文件并保存时，流程是这样的：

1. **文件监听（Watcher）**：构建工具监听文件系统变化（`fs.watch`）。
2. **增量编译**：
   - Webpack 只重新编译**修改过的那个模块**及其依赖，而不是全部重新打包。
   - Vite 直接重新编译修改的那个文件，并通知浏览器。
3. **WebSocket 通信**：开发服务器通过 WebSocket 发送“更新指令”给浏览器。
4. **原地替换**：浏览器接收到新模块，不刷新页面，直接在内存中替换掉旧的组件代码。**最厉害的是**：Vue 组件能保留当前组件的内部状态（如表单输入内容、页面滚动位置），只更新变动的那部分 DOM。

------

### 阶段四：开发调试增强（Dev Tools）

为了让开发者“爽”，Dev 环境会注入很多 Build 环境没有的辅助功能：

- **Source Maps（源码映射）**：生成最详细的 `source-map`，让你在浏览器开发者工具（F12）里打断点时，看到的不是转译后乱糟糟的代码，而是你写的原汁原味的 `.vue` 文件源码。
- **错误遮罩（Error Overlay）**：如果你的代码语法出错，页面不会白屏，而是在页面上浮出一个清晰的错误弹窗，直接告诉你哪一行报错（比如组件 `<template>` 里的语法错误）。

------

### 直观对比：Dev 环境 vs Build 环境

为了帮你彻底理清，我做了一个“打包策略”对比表：

| 维度           | npm run dev (开发)                            | npm run build (生产)                 |
| :------------- | :-------------------------------------------- | :----------------------------------- |
| **核心任务**   | 启动服务器 + 快速响应变化                     | 压缩、优化、分割代码                 |
| **输出位置**   | **内存（Memory）**，不生成 `dist` 文件夹      | **硬盘（Disk）**，生成 `dist` 文件夹 |
| **代码压缩**   | ❌ 不压缩（保留可读性）                        | ✅ 极致压缩（变量混淆）               |
| **死代码消除** | ❌ 保留（防止 HMR 失效）                       | ✅ 删除（Tree Shaking）               |
| **Source Map** | 最全（`eval-cheap-module`）                   | 极简（`none` 或 `hidden`）           |
| **启动速度**   | 随项目体积变大而变慢（Webpack）/ 极快（Vite） | 随项目体积变大而变慢（两者差距缩小） |

------

### 特别提醒：关于 `process.env.NODE_ENV`

虽然此时环境变量被设为 `"development"`，但请注意：

- 你在代码里写 `if (process.env.NODE_ENV === 'development') { console.log('debug') }`，构建工具会在此时**替换**为 `if ('development' === 'development')`。
- 这意味着，`console.log` 在 Dev 环境下**会被保留**，但在 Build 环境下因为被替换为 `if ('production' === 'development')`，会被判断为 `false` 并**彻底删除**。
