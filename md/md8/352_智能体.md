# 智能体

## 是什么？

**Agent（智能体） = 会自己动脑思考 + 会动手使用工具 + 能反复试错直到把事办成的人工智能。**

可以把传统的大模型（比如纯对话的ChatGPT）想象成一个 **“只有嘴和脑子的超级学霸”**。你问它一道数学题，它吧啦吧啦把解题步骤写给你，但最后算错了一个数字，它自己也不知道，因为**它只说一次，不检查**。

而**Agent**，是给这个学霸**装上了手、脚、计算器和纠错本**，并告诉他：“去，把这道题做对，做完检查，错了重来。”

------

**拆开来看，Agent有三大“超能力”：**

1. **拥有“灵魂”（自主规划）**：你给它一个模糊的大目标，比如“帮我策划下个月去北京玩”。它不会直接甩给你一个现成的攻略，而是会自己在脑子里列步骤：“第一步，先查机票价格；第二步，根据预算定酒店；第三步，排景点路线……”它会自己把大任务拆解成小任务。
2. **拥有“手脚”（工具调用）**：传统模型查不了实时天气，也订不了票。但Agent会主动去“抓”工具来用。它发现要查价格，就会自动去调取网上的机票API（接口）；要算预算，它就自己打开“计算器”工具；要记行程，它就调出“日历”插件。
3. **拥有“复盘脑”（反思迭代）**：这是最关键的一点！Agent做完一步，会回头看一眼结果。假如它订酒店时发现预算超了，它不会硬着头皮继续，而是会**自我纠正**：“哦，超预算了，那我重新筛选一个便宜点的，再试一次。”它会像人一样，**行动→观察结果→调整行动→再行动**，直到目标达成。

| 场景       | 传统大模型（LLM）                                            | 智能体（Agent）                                              |
| :--------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **写代码** | 一次性生成几百行代码给你，里面有个Bug，你得自己找出来告诉它。 | 写完代码后，自己运行测试，发现报错，自动分析报错日志，修改代码，再跑一遍，直到通过测试，才把最终版交给你。 |
| **查天气** | 它说：“我没法实时联网，请打开天气App看一下。”                | 它说：“好的，我正在调用天气插件……查到了，明天北京下雨，记得带伞，我已经帮你把雨伞加进待办清单了。” |
| **买咖啡** | 给你一份“附近咖啡馆Top10”列表。                              | 它说：“我检测到你上次喝的是拿铁，现在打开外卖平台，帮你下单了那家你常喝的店，预计15分钟后送到，这是取餐码。” |

------

**Agent的内核叫“ReAct框架”：**

- **Reason（推理）**：思考下一步该干嘛。
- **Act（行动）**：调用工具或执行命令。
- **观察结果**：看这一步做得对不对，然后循环回去继续Reason。

所以，**千万别把Agent当成“更聪明的聊天机器人”**，它是“**能独立干活儿的数字员工**”。以前你让AI干活，得一步步教它（写Prompt）。现在你只需要告诉Agent**“要什么结果”**，至于**“怎么干”**，它自己会边想边干边调整，最后直接给你交差。

LLM像“顾问”，只出方案不干活；Agent像“实习生”，拿上工具自己干，干错自己改，改完交结果。



## 如何实现？

Agent的实现，本质上就是在一个**“永不停歇的循环”**里运行。这个循环在专业上叫 **“感知-决策-执行-反馈”闭环**，代码层面就是一个 **while 循环**（直到任务完成才停止）。

------

### 第一层：大脑（核心模型 + 系统提示词）

这是Agent的“总司令部”。实现它，工程师会在发给大模型的**系统提示词**里，悄悄塞进一本**“员工手册”**。手册里写着：

- “你是实习生，你的目标是帮用户搞定事情。”
- “你有这些工具可以用（下面会讲）。”
- **最重要的紧箍咒**：“你必须按照‘思考→行动→观察’的格式回复，不允许只回答一次就结束。”

当你说“订酒店”时，大脑不是直接给答案，而是强制自己先输出思考草稿：“我在想，第一步应该先查预算……”

------

### 第二层：手脚（工具库 Tool Registry）

这是Agent能干活的关键。实现上，工程师会提前写好一堆**“工具说明书”**（API接口），挂在Agent的腰带上。每份说明书上都写着：

- 工具名字（比如 `search_hotel`）
- 需要什么参数（比如 `city` 城市, `price` 价格）
- 这个工具是干嘛用的

**关键实现细节**：现在的编程框架（比如LangChain、AutoGen）会让大模型**输出一段特定的JSON（一种结构化数据格式）**，而不是普通的人话。比如模型输出：

> ```
> {"tool": "search_hotel", "params": {"city": "北京", "price": 500}}
> ```

系统看到这段JSON，就知道**“哦，大脑决定调用订房工具了”**，于是立刻去执行这段代码。

------

### 第三层：记忆（短期 + 长期存储）

为了让实习生不“失忆”，工程师给它装了两种记忆：

- **短期记忆（上下文窗口）**：就是把“你刚才说的话”、“它上一步查到的结果”全部塞回给大模型。这就像给实习生一个**小黑板**，每做一步，就把这步的结果写在黑板上，让大脑随时看到。
- **长期记忆（向量数据库）**：如果你们聊了很多次，或者要查公司内部几千页的文档，小黑板写不下。这时候会把信息压缩成数字存进**外部数据库**，需要时再快速检索出来。这就相当于给实习生配了个**外接硬盘**。

------

### 第四层：刹车与方向盘（循环控制 + 终止条件）

如果没这层，Agent会**死循环**（比如一直订房退房订房退房）。所以工程师必须设定**“终止开关”**：

- **最大步数**：比如限制只能循环 10 步，超过就强制停止，报告“我尽力了”。
- **最终答案标识**：要求大脑在觉得搞定时，必须输出 `{"final_answer": "酒店已订好，是XX房"}`。程序一检测到这个标识，立刻跳出循环，把这句话返给你。

### 

假设你要订酒店，Agent后台实际在**飞速跑这个循环**：

| 循环次数  | 后台在干什么（伪代码逻辑）                                   | 相当于实习生                                 |
| :-------- | :----------------------------------------------------------- | :------------------------------------------- |
| **第1圈** | 把“订北京酒店”+ 手册 丢给大模型。 模型思考后输出JSON：`调用查价工具` | 实习生看任务，掏出手机查价格。               |
| **第2圈** | 系统执行查价代码，拿到结果`“500元/晚”`。 把**这个结果**连同上一次对话，重新丢给大模型。 模型再思考，输出JSON：`调用下单工具` | 实习生把价格写在黑板上，看了一眼，决定下单。 |
| **第3圈** | 系统执行下单代码，返回`“下单成功，订单号A”`。 再丢给大模型。 模型思考后输出JSON：`{"final_answer": "搞定了"}` | 实习生看到成功提示，向你汇报。               |
| **停止**  | 系统检测到`final_answer`，**跳出循环**，把答案发给你。       | 下班！                                       |

------


现在实现Agent，**根本不需要你从0写这个循环**。市面上有现成的框架（比如Python的 **LangChain** 或 **AutoGen**），你只需要像搭积木一样：

1. 选择用哪个“大脑”（GPT还是国产模型）。
2. 把“工具说明书”（API）填进框架的列表里。
3. 写一句启动命令：“开始干活！”

剩下的那个`while`循环，框架会自动替你跑起来。

------

现在你明白了吧？**实现Agent，就是写一个“会反复调用大模型”的程序**。大模型负责“想”，程序负责“干”，循环负责“直到干对为止”。



## LangChain

Agent的核心就是那个反复“思考-行动-观察”的循环。LangChain的巧妙之处在于，它把这个复杂的循环过程，封装进了一个“黑盒”里。

开发者要做的，其实就是 **“两行代码”**：**第一行定义工具，第二行创建并运行Agent**。

------

### 核心：把循环装进“黑盒”

LangChain（特别是1.0版本后的`create_agent`方法）将Agent循环封装成了一个**图（Graph）**。它的工作流程是这样的：

1. **调用模型**：Agent节点带着你的问题去问大模型。
2. **检查指令**：看模型返回的是不是“工具调用指令”（`tool_calls`）。
3. **执行工具**：如果是，就立刻执行对应的工具。
4. **返回结果**：把工具执行的结果（`ToolMessage`）再传回给模型。
5. **循环判断**：模型会再次思考，决定是继续调用工具，还是给出最终答案。

这个循环会一直持续，直到模型认为任务完成，不再发出新的工具调用指令为止。整个过程由Agent框架自动维护。

------

### 实战：“两行代码”搭建一个天气助手

用LangChain的`create_agent`方法，创建一个能查天气的Agent，核心代码非常简短。

**第一步：定义工具**
告诉Agent它能用什么“手脚”。

```python
# 1. 定义一个查天气的“工具”
def check_weather(location: str) -> str:
    """查询指定位置的天气。"""
    # 这里可以换成真实的API调用
    return f"{location} 的天气是晴朗的。"

# 把普通函数包装成LangChain的工具（在最新版本中，可以直接传入函数）
# 参考: https://reference.langchain.com/python/langchain/agents/factory/create_agent
```

**第二步：创建并运行Agent**
把“大脑”（模型）和“手脚”（工具）交给`create_agent`，然后让它干活。

```python
from langchain.agents import create_agent
# 假设已配置好模型, 例如: model = "openai:gpt-4o"

# 2. 创建Agent，把“大脑”和“手脚”组装起来
agent = create_agent(
    model="openai:gpt-4o",   # 指定大模型作为“大脑”
    tools=[check_weather],    # 把“查天气”这个工具交给它
    system_prompt="你是一个有帮助的助手。" # 给它一个“人设”
)

# 然后，让Agent运行起来
# agent.invoke() 内部会自动执行我们上面说的那个循环
result = agent.invoke(
    {"messages": [{"role": "user", "content": "旧金山的天气怎么样？"}]}
)
print(result)
```

当运行这段代码时，`agent.invoke()`内部就会自动启动那个“思考-行动-观察”的循环，直到它查到天气并回答你。

> **注**：`create_agent`是LangChain 1.0版本后的推荐用法。在更早的版本中，可能会看到`initialize_agent`或`AgentExecutor`等，但核心思想都是封装循环。

------

### “黑盒”里的秘密：执行器与中间件

这个“黑盒”之所以强大，是因为内部有两大法宝：

- **AgentExecutor（执行器）**：这是驱动循环的引擎。它负责在“大脑”和“手脚”之间不断传递信息，并时刻检查是否该停止了。
- **Middleware（中间件）**：这就像给Agent加的“插件”。你可以在循环的**前、中、后**各个节点插入自定义功能，比如：在调用模型前记录日志、在工具执行后检查结果、在Agent“跑偏”时及时纠正等等。

所以，**“两行代码”只是一个形象的说法**，它背后是LangChain框架为我们做的大量封装工作。我们作为开发者，只需要定义好工具，框架就会帮我们管理好那个复杂的决策循环。



## Skill & Tool

很多初学者会把这两个词混用，但在大模型应用开发（尤其是LangChain框架）里，**它们是“执行层”和“策略层”的严格区分**。

我用一个**“米其林后厨”**的比喻，帮你一秒分清：

- **Tool（工具）**：是后厨里的**“具体厨具”**——比如一把刀、一口锅、一个烤箱。它只管**“怎么干”**，动作极其单一、原子化。
- **Skill（技能）**：是厨师**“脑子里会的菜谱”**——比如“做一道红烧肉”。它管**“先干哪个、再干哪个”**，是把多个Tool按顺序组合起来的**标准化流程**。

------

### 1. Tool（工具）：原子化的“手脚”

在代码层面，**Tool就是一个被包装好的函数（Function）**。它必须有三个明确的特征：

- **名字**（叫什么）
- **描述**（什么时候用它，大模型靠这个描述来判断要不要调用）
- **输入/输出**（需要什么参数，返回什么结果）

**关键特性**：Tool **没有决策权**，它只负责**“执行”**。大模型让它“切菜”，它绝不会自作主张“顺便把火点上”。

**代码例子（LangChain写法）：**

```python
from langchain.tools import tool

# 这就是一个Tool——只负责把两个数加起来，别的啥也不管
@tool
def add_numbers(a: int, b: int) -> int:
    """当需要进行加法运算时，使用这个工具。"""
    return a + b

# 大模型调用它时，只会传(3, 5)，它返回8
```

------

### 2. Skill（技能）：封装好的“战术流程”

在LangChain最新生态（尤其是LangGraph）里，**Skill不是一个单一的函数，而是一个“子任务图”（Sub-graph）或“链”（Chain）**。

它的核心作用是：**把多个Tool按固定顺序绑在一起，形成一套成熟的“套路”**，让大模型不需要每次都重新规划细节。

**关键特性**：Skill **拥有逻辑判断**，内部可以包含“如果A则走Tool1，如果B则走Tool2”的分支。

**还是做红烧肉的例子：**

- **Tool**：`切肉()`、`烧水()`、`倒酱油()`、`定时关火()`
- **Skill（"做红烧肉"）**：内部写死了流程 -> `切肉()` -> `烧水焯肉()` -> `起锅烧油()` -> `倒酱油()` -> `小火炖40分钟()` -> `大火收汁()`。大模型只需要说“做道红烧肉”，Skill就会**自动按顺序调用这一串Tool**。

**代码例子（LangGraph写法，伪逻辑）：**

```python
# Skill本质是一个预编译好的子图
def cook_red_braised_pork(state):
    # 内部固定好了执行顺序，不用大模型每步都思考
    cut_meat()      # Tool1
    boil_water()    # Tool2
    add_sauce()     # Tool3
    # ...
    return "成品"
```

------

### 3. 最核心的区别对比表

| 对比维度     | **Tool（工具）**                     | **Skill（技能）**                                            |
| :----------- | :----------------------------------- | :----------------------------------------------------------- |
| **抽象层级** | 低（细粒度）                         | 高（粗粒度）                                                 |
| **本质**     | 单个函数（Function）                 | 一个工作流/子流程（Workflow / Sub-graph）                    |
| **决策权**   | **无**，指哪打哪                     | **有内部逻辑**，内部可判断先做A还是先做B                     |
| **谁来触发** | 由大模型（Agent）直接选              | 由大模型（Agent）整体调用，或由父Skill内部触发               |
| **能否拆分** | 不可再分（原子操作）                 | 可以拆分为多个Tool的组合                                     |
| **举例子**   | `查数据库()`、`发邮件()`、`加数字()` | `生成周报`（含查数据+画图+发邮件）、`预定出差行程`（含查机票+订酒店+写日程） |

------

### 4. 为什么LangChain要强行区分这俩？

因为**成本和效率**！

- **如果只用Tool**：大模型每次都要思考“先调用哪个Tool？再调用哪个？”，思考次数多（Token消耗大），且容易出错（顺序搞乱）。
- **如果把常用套路封装成Skill**：大模型只需要**一次调用**“生成周报”这个Skill，框架内部自动跑完10个Tool。**Token消耗骤减，稳定性大增**。

**总结：**

> **Tool是给Agent用的“原子级零件”，Skill是工程师提前焊好的“标准化模块”。Agent负责决定“用哪个模块”（Skill），Skill负责指挥“怎么拧零件”（Tool）。**



## MCP

MCP（Model Context Protocol，模型上下文协议），是一套**让AI的“大脑”能即插即用地连接各种“手脚”和“数据源”的通用标准**。

### 理解MCP：AI世界的“USB-C接口”

可以把MCP想象成**AI领域的“USB-C接口”**。

- **没有MCP之前**：每个AI应用（如Claude、ChatGPT）想连接一个外部工具（如数据库、日历），都需要开发者单独写一套“驱动”。每多一个工具，就得多写一套代码，非常繁琐。
- **有了MCP之后**：AI应用和被连接的工具、数据源都遵循MCP这个通用标准。任何支持MCP的AI应用，都能像U盘插上电脑一样，直接使用任何支持MCP的工具。

### MCP如何工作？一个“主机-客户端-服务器”的架构

MCP的精妙之处在于它的**客户端-服务器架构**，把不同的角色分得很清楚：

- **MCP主机 (Host)**：这是你直接交互的AI应用程序，比如**Claude Desktop**、**ChatGPT**或**VS Code**。它是整个流程的“指挥官”。
- **MCP客户端 (Client)**：它运行在主机内部，是主机与外界沟通的“信使”。每个客户端专门负责与一个MCP服务器保持一对一沟通。
- **MCP服务器 (Server)**：这是一个轻量级的程序，是真正的“干活的人”。它暴露特定的“能力”（也就是我们之前聊过的Tool），比如查询数据库、发送邮件等。一个主机可以连接多个服务器，每个服务器专注于提供某一种或几种能力。

### MCP服务器的核心能力：“三板斧”

一个MCP服务器主要通过三种方式为AI提供能力：

1. **资源 (Resources)**：提供**数据**，比如文件内容、数据库记录等。
2. **工具 (Tools)**：提供**可执行的函数**，让AI能对世界产生影响。
3. **提示词 (Prompts)**：提供**可复用的模板**，引导AI进行特定类型的对话或完成特定工作流。

### 总结：为什么MCP如此重要？

MCP的价值在于**标准化**和**解耦**。

- **对开发者**：只需按MCP标准开发一次服务器，就能被所有支持MCP的AI应用使用，极大减少了重复劳动。
- **对AI应用**：通过MCP可以接入一个不断增长的“能力生态”，快速获得新技能。
- **对整个行业**：它避免了各自为政的“接口大战”，让AI能力的共享和复用变得前所未有的简单。

所以，MCP就像是给AI世界制定了一套通用的“插头”标准，让数据和工具的连接变得像插拔U盘一样简单、高效。
