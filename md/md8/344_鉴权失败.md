# 鉴权失败

处理Web应用鉴权失败，不能只简单地返回一个“403”或“401”错误码，而需要**后端安全策略**、**前端用户体验**和**令牌生命周期管理**三位一体协同处理。

以下是标准的处理范式，按场景分层解析：

### 1. 核心区分：认证失败 vs 授权失败（后端首要逻辑）

后端必须精确区分这两种情况，并返回不同的HTTP状态码，这是前端正确处理的基础：

- **401 Unauthorized（认证失败）**：用户**未登录**或**Token无效/过期**。解决路径：重新登录。
- **403 Forbidden（授权失败）**：用户**已登录**，但**没有权限**访问该资源（如普通用户访问管理员后台）。解决路径：提示无权访问，而不是跳转登录。

------

### 2. 前后端交互的标准流程（Token过期场景）

这是最复杂的场景，以JWT（JSON Web Token）为例，推荐采用**“静默刷新”**机制：

1. **发起请求**：前端携带过期的Access Token请求接口。
2. **后端返回**：返回 `401` 状态码，且 `WWW-Authenticate` 头信息中包含 `error="invalid_token"`。
3. **前端拦截**：前端Axios/Fetch拦截器捕获到 `401`。
4. **尝试刷新**：前端**立即**调用刷新令牌（Refresh Token）接口（通常携带HttpOnly的Cookie或请求体中的Refresh Token）。
   - **刷新成功**：用新Token重试**原队列**中的所有等待请求（用户无感知）。
   - **刷新失败（Refresh Token也过期）**：**立即**清空本地存储，跳转至登录页，并携带 `redirect` 参数指向当前页面（以便登录后回跳）。
5. **防并发处理**：如果在刷新Token期间，页面同时发出了多个请求，需要将后续请求挂起（放入队列），待刷新成功后统一重发，避免触发多次刷新请求。

------

### 3. 前端用户体验（UI层面的处理）

不要在用户操作中途突然粗暴跳转，这会毁掉用户状态：

- **登录页跳转**：仅当**Refresh Token过期**或**用户主动登出**时，才清空Store并跳转。跳转前应使用`Message`提示具体原因（如“登录超时，请重新登录”），而不是直接空白页。
- **权限不足（403）**：**不要跳转登录页**。应弹出提示框“您暂无此操作权限”，或直接隐藏/禁用无权限的按钮（权限前置控制）。
- **静默处理**：对于非用户主动触发的轮询接口（如消息未读数），若鉴权失败，直接停止轮询并注销，而不是弹窗骚扰用户。

------

### 4. 安全层面的关键注意事项（必看）

处理失败时，漏洞往往出现在细节里：

- **区分错误提示**：登录接口返回“用户名或密码错误”（统一模糊提示），**严禁**明确提示“用户名不存在”或“密码错误”，防止账号枚举攻击。
- **记录审计日志**：后端在返回鉴权失败时，**必须**记录包含 `IP`、`User-Agent`、`请求路径` 的失败日志（WARN级别），用于异常行为检测（暴力破解预警）。
- **避免重定向循环**：确保登录页、静态资源（CSS/JS）和公开API（如 `/health`）**不校验**Token，否则页面加载时反复触发鉴权，导致死循环崩溃。
- **清除本地状态**：鉴权失败跳转登录页时，务必调用 `localStorage.clear()` 或清除特定Store，防止旧用户数据残留在内存中被下一个登录用户看到（信息泄露）。

------

### 5. 标准代码伪示例（前端拦截器）

```javascript
// 请求拦截器核心逻辑
let isRefreshing = false;
let failedQueue = [];

api.interceptors.response.use(
  response => response,
  async (error) => {
    const originalRequest = error.config;
    
    // 防止死循环：如果刷新接口本身401，直接放弃
    if (error.response?.status === 401 && !originalRequest._retry) {
       if (isRefreshing) {
          // 挂起请求，排队等待刷新完成
          return new Promise(resolve => failedQueue.push(resolve));
       }
       
       originalRequest._retry = true;
       isRefreshing = true;
       
       try {
          const newToken = await refreshTokenAPI();
          updateToken(newToken);
          // 重试队列中的所有请求
          failedQueue.forEach(cb => cb(api(originalRequest)));
          return api(originalRequest); // 重试当前请求
       } catch (refreshError) {
          // 刷新失败，清空状态跳转登录
          logoutAndRedirect();
          return Promise.reject(refreshError);
       } finally {
          isRefreshing = false;
          failedQueue = [];
       }
    }
    return Promise.reject(error);
  }
);
```

### 总结建议

**鉴权失败的最高处理原则是：用户无感知时尽量“静默复活”（刷新Token），用户操作时尽量“精准反馈”（403弹窗，401跳转），且全程不给攻击者提供“用户名有效性”的试探机会。**

如果你的项目尚未实现Refresh Token机制，建议优先升级；如果只是简单内部系统，则直接在401时跳转登录页并清空Store即可，不必过度设计。
