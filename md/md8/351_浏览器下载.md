# 浏览器下载

```javascript
const link = document.createElement('a')
link.style.display = 'none'
link.href = URL.createObjectURL(blob)
link.download = '重点问题详情.xlsx'
document.body.appendChild(link)
link.click()
document.body.removeChild(link)
URL.revokeObjectURL(link.href)
```

这段代码能实现下载，核心原理在于**将浏览器内存中的二进制数据（Blob）伪装成一个可访问的网络资源，然后利用HTML <a> 标签的默认导航行为，通过download属性强行将其转化为“保存文件”操作**。

### 1. 创建“内存中的虚拟文件地址” (`URL.createObjectURL`)

- **原理**：`blob` 对象只是浏览器内存中的一堆二进制数据（比如Excel的字节流），它本身并没有网络地址，浏览器无法直接“访问”它。
- **做了什么**：`URL.createObjectURL(blob)` 会在浏览器内部创建一个**唯一的引用映射**，并生成一个特殊的 `blob:https://...` 链接。这个链接就像一把“钥匙”，指向内存中的那堆数据。浏览器看到这个协议，就知道不需要去网络请求，直接从内存读取即可。

### 2. 利用 `<a>` 标签的导航机制

- **原理**：点击一个带有 `href` 属性的 `<a>` 标签，浏览器的默认行为是**页面跳转/导航**（比如从当前页跳到 `https://xxx.com`）。
- **做了什么**：当 `link.href` 设为那个 `blob:` 地址后，`link.click()` 触发了一次“导航请求”。但这次导航的目标不是远程服务器，而是浏览器自己的内存区域。

### 3. 用 `download` 属性劫持导航行为

- **原理**：这是最关键的一步。普通的 `<a>` 跳转，如果指向的是图片、PDF或Excel文件，浏览器通常会**直接在新标签页打开并渲染**（预览），而不是保存。
- **做了什么**：`download` 属性强制告诉浏览器：“**不要尝试预览或跳转，请把这个请求的返回内容当作附件（Attachment）处理，直接弹出‘另存为’对话框，并用我指定的名字保存。**” 

### 4. 程序化触发用户行为 (`click()`)

- **原理**：浏览器的安全策略通常禁止脚本在未经用户手势（如点击按钮）触发的情况下，自动弹出下载窗口（防止恶意广告或病毒）。
- **做了什么**：虽然 `link.click()` 是代码触发的，但它通常必须放在用户真实点击事件的回调函数中执行（比如按钮的 `onClick`）。浏览器会校验调用栈，只要是由合法手势衍生出的同步代码，下载就会被放行。

------

### 底层浏览器机制（宏观视角）

> 用户点击按钮 → JS生成Excel数据(Blob) → 创建Object URL(内存映射) → 触发a标签点击 → 浏览器导航模块接收请求，识别`blob:`协议 → 读取内存数据 → 检测到`download`属性 → 将数据流交给文件管理器（File Manager）→ 弹出系统保存对话框 → 写入磁盘。

------

### 关于最后的清理 (`URL.revokeObjectURL`)

- **原理**：创建 `Object URL` 时，浏览器会把该 `Blob` 对象**锁定在内存**中，并计入引用计数。如果不主动释放，即使下载完了，这块内存依然被占用，会导致**内存泄漏**。
- **为什么要立即释放**：因为 `link.click()` 是**同步**触发的，一旦调用，浏览器已经把数据流交给了下载管理器（下载任务已开始）。此时立即调用 `revokeObjectURL` 切断映射，**不会**中断正在进行的下载任务，但能立刻告诉垃圾回收机制“这块内存可以回收了”，实现了高效的内存管理。

------

### 一个容易被忽略的细节

如果去掉 `link.download` 属性，这段代码在 Chrome 中会直接在新标签页打开一个包含乱码的二进制页面（因为浏览器试图渲染Excel为文本），或者只预览图片。而加上 `download`，就完成了从“预览”到“下载”的华丽转身。这就是这段代码能够实现下载的根本支点。
