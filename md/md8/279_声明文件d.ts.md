# 声明文件 d.ts

## 声明文件 declare

当使用第三方库时，需要引用它的声明文件，才能获得对应的代码补全、接口提示等功能。

```ts
declare var 声明全局变量
declare function 声明全局方法
declare class 声明全局类
declare enum 声明全局枚举类型
declare namespace 声明（含有子属性的）全局对象
interface 和 type 声明全局类型
/// <reference /> 三斜线指令
```



```ts
declare module 'express' {
    interface Router {
        get(path: string, cb: (req: any, res: any) => void): void;
    }
    interface App {
        use(path: string, router: any): void;
        listen(port: number, cb?: () => void): void;
    }
    interface Express {
        (): App;
    	Router(): Router;
    }
	const express: Express;
	export default express;
}
```



在 TypeScript 中，`declare` 关键字的核心作用是**环境声明**（Ambient Declaration）。它的本质是告诉 TypeScript 编译器：“**这个变量/函数/类/模块是存在的，但它的定义不在当前 TypeScript 代码中（通常由外部 JavaScript 环境、浏览器 API 或第三方库提供），请相信我，不要报错，并在编译时保留这些类型信息。**”

`declare` 编译后不会生成任何 JavaScript 代码，它只用于编译时的类型检查。

以下是 `declare` 的几种核心用法及实战场景：

### 1. 声明全局变量/函数/类（用于挂载在 `window` 或 `global` 上的变量）

当你在浏览器中通过 `<script>` 标签引入了某个 JS 库，或者在后端 Node.js 中挂载了全局对象时，使用 `declare` 来告诉 TS 它们的形状。

```typescript
// 声明一个全局变量
declare const API_URL: string;
console.log(API_URL); // 不会报错

// 声明一个全局函数
declare function greet(name: string): string;

// 声明一个全局类
declare class Person {
  constructor(name: string);
  sayHello(): void;
}

// 典型场景：补充 window 上的自定义属性
declare interface Window {
  __INITIAL_STATE__: any; // 比如 SSR 时挂载到 window 上的数据
}
// 此时 window.__INITIAL_STATE__ 就可以安全访问了
```

### 2. 声明模块（`declare module`）—— 处理无类型定义的第三方包

这是日常开发中最常用的场景。当 npm 包没有提供 `.d.ts` 类型文件，又无法及时补充完整类型时，用 `declare module` 来“忽略”类型检查。

```typescript
// 在 .d.ts 或 .ts 文件中
declare module "some-old-library" {
  export function init(config: any): void;
  export const version: string;
}
// 现在 import * as lib from "some-old-library"; 就不会报错了

// 通配符模块声明（常用于处理非 JS 资源，如 Vue 单文件、CSS、图片）
declare module "*.vue" {
  import type { DefineComponent } from "vue";
  const component: DefineComponent<{}, {}, any>;
  export default component;
}

declare module "*.css"; // 让 import './index.css' 不报错
declare module "*.png" {
  const src: string;
  export default src;
}
```

### 3. 声明命名空间（`declare namespace`）

用于描述通过全局 `<script>` 引入的、挂载在全局对象下的**嵌套层级较深**的库（例如 JQuery 的 `$` 或 `_`）。Namespace 主要解决的是遗留 JS 库的层级引用问题。

```typescript
declare namespace MyLib {
  interface Config {
    endpoint: string;
  }
  function request(url: string, config: Config): Promise<any>;
  namespace Utils {
    function format(data: any): string;
  }
}

// 使用：
MyLib.request('/api', { endpoint: 'v1' });
MyLib.Utils.format({});
```

### 4. 声明类、枚举、类型别名（仅限 `.d.ts` 环境）

在声明文件（`.d.ts`）中，`declare` 通常可以省略，因为 `.d.ts` 里的顶层 `interface` 和 `type` 默认就是全局声明。但为了显式表达，仍可加上。

```typescript
declare type Status = 'pending' | 'success' | 'error';
declare enum Color { Red, Green, Blue }
```

### 5. 扩展全局作用域（`declare global`）—— 非常重要！

当你在 **模块文件**（即有 `import` 或 `export` 的文件）中定义全局类型时，必须用 `declare global` 包裹，否则 TS 会认为该类型仅在当前模块作用域有效。

```typescript
// 在 utils/global.d.ts 或某个 .ts 文件中
export {}; // 必须有 export/import，将其变为模块

declare global {
  interface String {
    customTrim(): string;
  }
  var myGlobalVar: number;
}

// 实现该原型方法（在 JS 运行时）
String.prototype.customTrim = function() {
  return this.replace(/ /g, '');
};
```

------

### 6. 进阶：声明合并（与 `declare` 配合）

`declare` 常与**声明合并**配合使用，用于给已有的第三方库或内置对象打补丁。

```typescript
// 给内置 Array 添加新方法类型
declare global {
  interface Array<T> {
    getLast(): T | undefined;
  }
}
// 运行时再挂载 Array.prototype.getLast ...
```

------

### 7. `declare` 与 `type` / `interface` 的区别

- **interface / type**：定义纯粹的类型，不涉及运行时值。在 `.ts` 中默认仅在当前文件有效（除非导出）。
- **declare**：**断言**该“实体”（变量/函数/类/模块）存在于当前运行时环境中，TS 会将其视为全局可用的环境变量。它通常放在 `.d.ts` 文件中，或配合 `export {}` + `declare global` 在普通 `.ts` 中使用。

------

### 总结与最佳实践

| 场景                       | 用法                                               |
| :------------------------- | :------------------------------------------------- |
| 引用 CDN 加载的旧 JS 库    | `declare var LibraryName` 或 `declare namespace`   |
| 安装的 npm 包没有类型      | 创建 `types/xxx.d.ts`，使用 `declare module "xxx"` |
| 导入图片、CSS、Vue 等资源  | 使用 `declare module "*.png"` 等通配符             |
| 给 `window` 添加自定义属性 | 使用 `declare interface Window { ... }`            |
| 在模块文件中定义全局类型   | 必须用 `declare global { ... }` 包裹               |

**注意**：`declare` 只能定义类型，不能包含具体的逻辑实现。如果你在 `declare function` 后直接写 `{ return ... }`，TS 会报错，因为 `declare` 区域禁止包含运行时代码。如果代码和类型都在同一个 `.ts` 文件中，请直接写 `function` 实现，无需加 `declare`。



```ts
/// <reference types="vite/client" />

declare module "*.vue" {
  import type { DefineComponent } from "vue";

  const component: DefineComponent<object, object, any>;
  export default component;
}

declare module "@/constants/menus.js";

interface Window {
  api: string;
  doc: string;
  llmApi: string;
  filePreviewUrl: string;
  loginAccountInfo: {
    username?: string;
    password?: string;
  };
}

declare module "vue-ls";
```

这段代码是 TypeScript 项目（通常是 Vue 3 + Vite 项目）中的**环境声明文件**（通常放在 `env.d.ts`、`shims-vue.d.ts` 或 `vite-env.d.ts` 中）。它的目的是**补全类型信息**，让 TypeScript 能够识别项目中特定的文件、全局变量和第三方库。

下面我为你逐块拆解，详细解释每一部分的含义和背后的作用：

### 1. `/// <reference types="vite/client" />`

**作用**：三斜线引用指令，告诉 TypeScript 编译器**加载 Vite 提供的官方客户端类型定义**。

- **包含的内容**：引入后，你可以在代码中直接使用 `import.meta.env` 及其智能提示（如 `import.meta.env.VITE_APP_TITLE`），并且让 Vite 支持导入 `.css`、`.svg`、`.png` 等静态资源时拥有正确的类型。
- **本质**：这是 Vite 项目的“标配”，没有这行，`import.meta.env` 会在 TS 中报红。

------

### 2. `declare module "*.vue" { ... }`

**作用**：通配符模块声明，告诉 TypeScript **如何处理 .vue 单文件组件的导入**。

- **具体逻辑**：当你在代码中写 `import HelloWorld from './HelloWorld.vue'` 时，TS 无法直接解析 `.vue` 文件。这段代码将所有的 `*.vue` 文件定义为一个 Vue 组件对象。
- **关于泛型 DefineComponent<object, object, any>**：
  - `DefineComponent` 是 Vue 内部定义的组件类型。
  - 三个泛型分别代表 **Props（属性）**、**Emits（事件）** 和 **Slots（插槽）**。
  - 这里填 `object` 和 `any` 是比较宽松的写法（意味着 Props 默认是对象，插槽为 any）。**更严谨的官方推荐写法**通常是 `DefineComponent<{}, {}, any>`（空对象表示无 Props），但你的代码使用 `object` 在大多数场景下也能正常工作，表示接受一个对象作为 Props 定义。

------

### 3. `declare module "@/constants/menus.js";`

**作用**：**单一模块声明**，用于解决特定 JS 文件缺少类型定义的问题。

- **背景**：你的项目里有一个 `@/constants/menus.js` 文件（注意后缀是 `.js`）。在 TS 项目中，直接导入 `.js` 文件且没有同名的 `.d.ts` 时，TS 会报错“无法找到模块”。
- **这里的处理**：这行代码粗暴地告诉 TS：“**这个文件是存在的，你把它当成 any 类型处理就行，别报错**”。
- **注意**：它没有定义导出内容的形状（比如 menus 是数组还是对象），所以导入后该变量会被隐式视为 `any`，失去了类型校验。这是一种“快速修复”手段，如果有时间，最好给 `menus.js` 补充具体的类型定义。

------

### 4. `interface Window { ... }`

**作用**：**接口声明合并（Declaration Merging）**，用于扩展浏览器全局对象 `Window` 的类型。

- **原理**：TypeScript 内置的 `Window` 接口定义在 `lib.dom.d.ts` 中。这里重新声明 `Window`，TS 会自动将这两个接口**合并**，从而允许你安全地访问挂载在 `window` 上的自定义属性。
- **具体字段解释**（这些通常是后端返回给前端的环境变量或配置）：
  - `api: string`：后端 API 基础地址（如 `https://api.example.com`）。
  - `doc: string`：文档服务地址。
  - `llmApi: string`：大语言模型接口地址。
  - `filePreviewUrl: string`：文件预览服务地址。
  - `loginAccountInfo: { username?: string; password?: string }`：登录账户信息（`?` 表示可选），可能用于页面跳转前的回显或单点登录传递。
- **运行时**：这些属性通常由后端渲染模板（如 EJS、JSP）或 `index.html` 中的 `<script>` 标签注入，例如 `window.api = 'xxx'`。

------

### 5. `declare module "vue-ls";`

**作用**：声明 `vue-ls`（一个 Vue 插件，用于操作 LocalStorage 和 SessionStorage）这个第三方 npm 包存在。

- **背景**：`vue-ls` 的老版本可能没有自带的 TypeScript 类型声明文件（或者没有安装 `@types/vue-ls`）。
- **效果**：加了这行后，`import VueLs from 'vue-ls'` 或 `app.use(VueLs)` 不会再报“模块未找到”的错误。同样，导入后的内容被默认为 `any` 类型。

------

### 总结与建议（避坑提醒）

| 代码片段                                | 解决的问题                  | 类型安全程度                        |
| :-------------------------------------- | :-------------------------- | :---------------------------------- |
| `/// <reference types="vite/client" />` | Vite 环境变量和静态资源导入 | **高**（官方标准）                  |
| `declare module "*.vue"`                | Vue SFC 组件导入            | **高**（但 `object` 可优化为 `{}`） |
| `declare module "@/constants/menus.js"` | 特定 JS 文件导入            | **极低**（等同于 `any`）            |
| `interface Window { ... }`              | 浏览器全局自定义变量        | **高**（定义了具体结构）            |
| `declare module "vue-ls"`               | 第三方无类型库导入          | **极低**（等同于 `any`）            |

**补充一个进阶建议**：对于 `Window` 接口中的 `loginAccountInfo`，既然包含 `password`，**务必确保该值只在首次登录或非明文传输场景下使用**，避免在前端将密码明文长期暴露在控制台或内存中。同时，如果后端返回的数据结构与这里不匹配，TS 会报错，这正好起到了**类型校验**的保护作用。
