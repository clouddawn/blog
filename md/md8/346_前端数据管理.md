# 前端数据管理

## 前端的数据分为几种？

前端数据的分类没有绝对唯一的标准，通常从**作用域、存储介质、数据来源**和**更新频率**四个维度来划分，能帮你更清晰地理解数据管理（比如什么时候该用 Pinia）。

### 1. 按“作用域与归属”划分

这是前端状态管理中最经典的分类方式，直接决定了数据放在哪里。

- **组件局部状态**：只在单个组件内部使用的数据。例如一个按钮的 `loading` 状态、弹窗的 `visible` 开关。通常使用 Vue 的 `ref` 或 `reactive` 管理，不需要 Pinia。
- **全局共享状态**：多个不相关的组件（或跨页面）需要共同使用的数据。例如用户登录信息、权限菜单、全局主题色、购物车数量。**这正是 Pinia 要解决的核心问题**，将这类数据抽离到 Store 中统一管理。
- **路由/页面状态**：存放在 URL 参数（query/params）或路由元信息（meta）中的数据。例如列表页的筛选条件、详情页的 ID。这类数据刷新页面不会丢失，通常与 `vue-router` 配合读取。

### 2. 按“存储介质与持久化”划分（涉及刷新是否丢失）

根据数据是否保存在浏览器硬盘上，决定了刷新页面后数据还在不在。

- **运行时内存状态**：存储在 JS 变量或 Pinia Store 中。页面刷新或关闭后，数据会**彻底消失**（重置）。这是最常见的数据形态，适合存放临时的交互数据。
- **持久化状态**：存储在浏览器的 LocalStorage、SessionStorage 或 IndexedDB 中。页面刷新或关闭后再次打开，数据**依然存在**。通常用于记住“记住我”的 Token、用户偏好设置。使用 Pinia 时，常配合 `pinia-plugin-persistedstate` 插件将部分 Store 数据自动同步到本地存储。
- **Cookie**：特殊的小文本数据，每次 HTTP 请求会自动携带，主要用于服务端身份验证。

### 3. 按“数据来源”划分（决定数据获取方式）

区分数据是前端自己生成的，还是从后端拿到的。

- **服务端数据**：通过 Ajax/Fetch 请求从后端 API 获取的数据（如用户列表、商品详情）。这类数据通常是**异步**的，需要处理加载态和错误态，在 Pinia 的 `actions` 中执行请求并存储结果。
- **前端静态数据**：写死在代码中的常量配置、枚举值、字典表（如性别选项 `[{label:'男', value:1}]`）。这类数据不需要请求，可以直接定义在组件外或独立的 `constants.ts` 文件中。
- **用户交互产生的即时数据**：用户输入表单时的临时值、鼠标拖拽产生的坐标。这类数据变动非常频繁，通常只存在于组件局部，不需要存入 Store 或数据库。

### 4. 按“更新频率与变化特性”划分（决定响应式策略）

- **静态/低频数据**：如系统配置、国家城市列表。一旦加载，很少变化。可以缓存起来，避免重复请求。
- **动态/高频数据**：如股票行情、倒计时、实时聊天消息。这类数据变化极快，需要高效的响应式更新。在 Vue 中需注意避免大规模频繁修改响应式数据造成性能开销（可使用 `shallowRef` 或 `markRaw` 优化）。

------

**总结一下在实际开发中的“黄金搭配”原则：**

1. **组件局部 + 临时交互** → 直接用 `ref`/`reactive`。
2. **多组件共享 + 异步请求** → 放在 **Pinia Store** 中管理（核心用法）。
3. **需要刷新页面不丢失（如 Token）** → Pinia Store **+ 持久化插件**，同步写入 LocalStorage。
4. **URL 可记录（如搜索关键词）** → 放在路由 `query` 中，便于分享链接。

## 什么样的数据需要持久化？

**如果用户刷新页面后，数据丢失会让他们感到“懊恼”或“不便”，并且数据本身不涉及极高安全风险，那么就值得持久化。**

### 1. 用户身份与鉴权凭证（最刚需）

这是持久化的**第一优先级**。如果不持久化，用户每刷新一次页面就要重新登录，产品根本无法使用。

- **典型数据**：`accessToken`（访问令牌）、`refreshToken`（刷新令牌）、用户ID。
- **存储方式**：通常存入 `localStorage` 或 `sessionStorage`（视安全策略而定），并在 Pinia 中配合持久化插件自动同步。

### 2. 用户个性化偏好与设置（体验类）

这类数据丢失不会导致功能不可用，但会让用户觉得“这个网站记不住我”，体验大打折扣。

- **典型数据**：**主题配色**（深色/浅色模式）、**语言/国际化**（中/英文）、**侧边栏折叠状态**、**表格每页显示条数**。
- **存储方式**：`localStorage` 或 Cookie。这类数据通常体积很小，适合永久保存在本地。

### 3. 高频使用的前端字典与配置（性能类）

这类数据变动频率极低（比如几天甚至几个月才变一次），如果每次进入页面都去后端请求，会浪费带宽并拖慢加载速度。

- **典型数据**：**省市区三级联动数据**、**系统枚举字典**（如订单状态 `0:待支付, 1:已支付`）、**静态产品分类树**。
- **存储方式**：`localStorage` 并加上**版本号或时间戳**。当后端数据更新时，通过版本号对比决定是否重新请求覆盖。

### 4. 用户未完成的草稿与操作进度（防丢失类）

用户在页面上花费了大量精力输入或操作，如果因刷新或误关页面导致数据清零，非常打击用户留存。

- **典型数据**：**富文本编辑器的草稿内容**、**长表单的已填部分**、**视频/音频的播放进度**。
- **存储方式**：`localStorage` 或 `IndexedDB`（数据较大时）。通常配合 `beforeunload` 事件自动保存。

------

### 什么时候“不需要”持久化？（避坑指南）

1. **敏感隐私数据（如密码、支付密码、密钥）**：绝不能存入 `localStorage`，因为 XSS 攻击可以轻易读取本地存储。应尽量存于内存（Pinia/Ref）或 HttpOnly Cookie 中。
2. **实时性要求高的数据（如股票行情、消息未读数）**：持久化没有意义，因为几秒后就过期了，刷新后必须请求最新数据。
3. **体积巨大的数据（如图片 Base64、大列表全量数据）**：`localStorage` 只有约 **5-10MB** 容量，且读写是同步的，存储大数据会阻塞页面渲染。大数据请走 `IndexedDB` 或内存缓存。

------

### 给你的“决策四连问”（实用小工具）

当你纠结某个数据要不要持久化时，问自己这四个问题：

1. **刷新页面后，用户是否期望它还在？**（是，则考虑持久化）
2. **这个数据是否涉及财产安全或隐私？**（是，则**禁止**放入 localStorage，改用内存或加密处理）
3. **这个数据是否经常被后端修改（秒/分钟级）？**（是，则**不持久化**，或设置极短的过期时间）
4. **如果不存，需要用户重新操作多少步骤？（少于 3 步？不存；多于 5 步？存！）**
