## 进程、线程

**进程（Process）** 是操作系统进行**资源分配**的基本单位。可以把它理解为**一个正在运行的程序实例**。

- 每个进程拥有**独立的内存空间**（代码、数据、堆栈），彼此隔离。
- 一个进程崩溃通常不会影响其他进程。
- 切换进程的开销较大（需要保存 CPU 状态、刷新缓存、切换内存映射等）。

**线程（Thread）** 是操作系统进行**调度执行**的基本单位。可以把它理解为**进程内部的一条执行流水线**。

- 一个进程可以包含多个线程，所有线程**共享**该进程的内存资源（全局变量、堆等），但每个线程拥有自己的栈和寄存器状态。
- 线程之间切换开销小，通信方便（直接读写共享内存），但一个线程出错可能导致整个进程崩溃。



- **进程 = 车间**（有独立的厂房、机器、材料）
- **线程 = 车间里的流水线**（共享车间的资源，各流水线独立运转）



## 操作系统

**操作系统（Operating System，简称 OS）** 是管理计算机硬件与软件资源的**核心系统软件**，同时也是用户与计算机硬件之间的**接口**。

### 简单理解

可以把操作系统看作计算机的**大管家**：

- 它负责**分配** CPU、内存、硬盘等硬件资源给各个程序。
- 它**隔离**程序的相互干扰（比如一个程序崩溃不会立刻让整个电脑死机）。
- 它提供**统一的服务**（比如文件读写、网络通信、用户界面），让软件开发者不用直接操作复杂的硬件。

### 主要功能

1. **进程管理**：创建、调度、终止进程和线程。
2. **内存管理**：分配和回收内存，确保各程序互不冲突。
3. **文件系统管理**：组织、存储、读写文件（例如 C 盘、桌面文件夹）。
4. **设备管理**：驱动键盘、鼠标、显示器、硬盘等硬件。
5. **用户接口**：提供命令行（Shell）或图形界面（GUI），让用户与计算机交互。

### 常见的操作系统

- **Windows**（个人电脑最流行）
- **Linux**（服务器、安卓底层）
- **macOS**（苹果电脑）
- **iOS / Android**（手机、平板）



## 内存

**内存（Memory）** 通常指**主存储器**（RAM，Random Access Memory），是计算机中用于**临时存储**正在运行的程序和正在处理的数据的硬件。

可以把内存想象成**工作台**：

- 你要修理东西，得把工具和零件从**仓库**（硬盘）搬到**工作台**（内存）上才能动手。
- 工作台面积有限，放不下所有东西；而且一旦断电，工作台上的东西就会**全部消失**。

### 核心特点

1. **临时性（易失性）**：断电后数据立即丢失。
2. **高速读写**：比硬盘（机械硬盘或固态硬盘）快得多，直接与 CPU 交互。
3. **容量相对较小**：普通电脑 8GB、16GB、32GB 等，而硬盘通常是 256GB、512GB、1TB 等。

### 与硬盘的区别

| 特性   | 内存（RAM）        | 硬盘（Storage）           |
| :----- | :----------------- | :------------------------ |
| 作用   | 运行程序和当前数据 | 长期保存文件、操作系统等  |
| 速度   | 极快               | 慢（即使 SSD 也比内存慢） |
| 容量   | 小（GB 级别）      | 大（几百 GB 到几 TB）     |
| 断电后 | 数据丢失           | 数据保留                  |
| 价格   | 每 GB 更贵         | 每 GB 更便宜              |



## 栈和堆

在 JavaScript 中，**栈（Stack）** 和 **堆（Heap）** 是内存中两个不同的区域，用于管理不同类型的数据。

### 栈（Stack）

- **特点**：**有序、大小固定、自动分配与释放**（后进先出 LIFO）。
- **存储内容**：**基本类型**（`number`、`string`、`boolean`、`null`、`undefined`、`symbol`、`bigint`）以及**引用类型的地址**（即指向堆中对象的指针）。
- **生命周期**：函数执行时，其局部变量被压入栈；函数执行完毕，自动弹出销毁。

### 堆（Heap）

- **特点**：**无序、大小动态、手动分配**（JS 引擎自动垃圾回收）。
- **存储内容**：**引用类型**（对象、数组、函数、`Date`、`RegExp` 等）的实际数据。
- **生命周期**：对象在堆中创建，不被任何引用时被垃圾回收器回收。

### 图解示例

```javascript
let name = "Alice";      // 基本类型 → 栈中直接存值 "Alice"
let age = 25;            // 基本类型 → 栈中直接存值 25

let person = {           // 对象 → 堆中存实际数据 { name: "Bob", age: 30 }
  name: "Bob",
  age: 30
};
// 栈中 person 变量存的是该对象的引用地址（如 0x1234）

let hobbies = ["coding", "reading"];  // 数组也是对象 → 堆中存数据，栈存引用
```



**内存示意图**：

```text
栈（Stack）                    堆（Heap）
+----------------+            +---------------------+
| name: "Alice"  |            | person: {           |
| age: 25        |            |   name: "Bob",      |
| person: 0x1234 |----------->|   age: 30           |
| hobbies: 0x5678|----+        | } (地址 0x1234)     |
+----------------+    |        +---------------------+
                      |        | ["coding","reading"]|
                      |        | (地址 0x5678)       |
                      +------->+---------------------+
```



### 关键区别（JS 视角）

| 特性         | 栈                     | 堆                               |
| :----------- | :--------------------- | :------------------------------- |
| 存储数据类型 | 基本类型 + 引用地址    | 引用类型的实际数据               |
| 大小         | 固定（每帧大小已知）   | 动态（对象可以任意扩展属性）     |
| 速度         | 访问更快               | 访问稍慢（需间接寻址）           |
| 内存管理     | 自动（函数结束即释放） | 垃圾回收（GC）自动回收           |
| 赋值行为     | 基本类型赋值拷贝值     | 对象赋值拷贝引用（共享同一对象） |

### 赋值与引用的经典例子

```javascript
let a = 10;        // 栈
let b = a;         // 栈：拷贝值，b = 10
b = 20;            // a 还是 10，互不影响

let obj1 = { x: 1 };   // obj1 在栈存引用，对象数据在堆
let obj2 = obj1;       // 拷贝引用，obj2 和 obj1 指向堆中同一对象
obj2.x = 2;
console.log(obj1.x);   // 2 —— 因为修改的是堆中的数据
```



### 函数调用时的栈变化

```javascript
function greet(name) {    // name 是基本类型，参数入栈
  let message = "Hello";  // 局部变量入栈
  return message + name;
}
greet("Alice");
// 调用时：参数 "Alice" 入栈 → 执行 → 函数结束，栈帧弹出
```



## 指针

**指针（Pointer）** 是一个变量，但它存储的不是普通数据（如数字、字符），而是**内存地址**——即另一个变量在内存中的位置。

可以简单理解为：指针就是**门牌号**。

- 普通变量是一个**房子**，里面住着值（比如数字 10）。
- 指针变量是一张**纸条**，上面写着某个房子的门牌号。
- 通过纸条上的门牌号，就能找到那个房子，然后查看或修改里面的东西。

### 直观例子（C 语言）

```c
int a = 42;      // 普通变量 a，内存中某个地址（比如 0x1000）存放 42
int *p = &a;     // 指针 p，存放 a 的地址（0x1000）
printf("%d", *p); // 通过 p 找到地址 0x1000，输出 42
*p = 99;         // 通过指针修改 a 的值，现在 a 变成 99
```

- `&a` 表示取 a 的地址。
- `*p` 表示“取 p 所指向的地址里的内容”。

### 为什么需要指针？

1. **间接访问**：函数内可以修改函数外的变量（C 语言中的“传址调用”）。
2. **动态内存分配**：`malloc` 返回的就是指向新内存块的指针。
3. **效率**：拷贝一个指针（4/8 字节）比拷贝一个很大的结构体快得多。
4. **数据结构的基石**：链表、树、图等都需要指针来串联节点。

### 指针的风险

- **野指针**：指针存了一个无效地址，访问会导致程序崩溃。
- **内存泄漏**：丢失了指向 `malloc` 内存的指针，导致那块内存永远无法释放。

### 与 JavaScript 的关系

JS 中没有显式的指针语法，但**对象引用**的本质就是指针（更安全，不能做指针运算）。

javascript

```
let obj = { value: 10 };
let ref = obj;   // ref 存的是堆中 obj 的地址（JS 引擎自动管理）
ref.value = 20;
console.log(obj.value); // 20 —— 因为 ref 和 obj 指向同一块内存
```



只不过 JS 隐藏了地址的概念，也避免了“野指针”和手动内存管理。



> **指针是一个存放内存地址的变量，通过它可以间接访问和操作另一块内存中的数据。**



## 垃圾回收

**垃圾回收（Garbage Collection, GC）** 是一种自动内存管理机制。它会自动找出程序中**不再使用的内存**（即不再被任何变量或对象引用的数据），并将其释放，以便后续重新利用。

在 JavaScript 中，开发者不需要像 C/C++ 那样手动 `malloc` 和 `free`；JS 引擎会定期执行垃圾回收，自动清理无用的内存。

### 核心原理：可达性（Reachability）

JS 内存管理的基本思想是**可达性**：从一组“根”（roots）对象出发，能通过引用或作用域链访问到的数据就是**可达的**（即仍在使用），反之则被认为是**不可达的**，可以被回收。

**根对象**包括：

- 全局对象（浏览器中的 `window`，Node.js 中的 `global`）
- 当前函数调用栈上的局部变量、参数
- 其他一些内置对象（如正在执行的闭包引用的外部变量）

### 主要算法：标记-清除（Mark-and-Sweep）

这是现代 JS 引擎最常用的垃圾回收算法。

1. **标记阶段**：从根对象出发，递归地遍历所有被引用的对象，并将其标记为“可达”。
2. **清除阶段**：遍历堆内存中的所有对象，清除那些没有被标记的对象（即不可达对象），回收它们占用的内存。

```javascript
// 示例1：简单情况
let person = { name: "Alice" };  // 对象 {name:"Alice"} 被变量 person 引用 → 可达
person = null;                    // 引用被切断，该对象不再可达 → 下一次 GC 会回收它
```



### 示例：引用计数（旧算法，已不单独使用，但可帮助理解陷阱）

早期的 JS 引擎使用引用计数：每个对象记录被引用的次数，次数为 0 则回收。
**问题**：循环引用会导致内存泄漏。

```javascript
function leak() {
  let objA = { name: "A" };
  let objB = { name: "B" };
  objA.ref = objB;   // A 引用 B
  objB.ref = objA;   // B 引用 A
  // 函数执行结束，objA、objB 变量出栈，但两个对象仍互相引用，引用计数不为 0
  // 引用计数算法无法回收，造成内存泄漏
}
leak();
```



现代 JS 引擎采用**标记-清除**，能从根对象出发检测到整个循环引用块不再被外部引用，从而正确回收。

### 示例：闭包中的可达性

```javascript
function outer() {
  let bigData = new Array(1000000).fill("*"); // 大数组
  return function inner() {
    console.log("hello"); // 内部函数没有使用 bigData
  };
}
let fn = outer(); // outer 执行后，bigData 本应被回收，但……
// 注意：inner 没有引用 bigData，但不同 JS 引擎对闭包的处理不同。
// 现代引擎（V8）会进行逃逸分析，如果闭包未使用外部变量，则不会保留 bigData。
// 更典型的泄漏场景：
function outer2() {
  let bigData = new Array(1000000).fill("*");
  return function inner() {
    return bigData.length; // inner 使用了 bigData
  };
}
let fn2 = outer2(); // 只要 fn2 存在，bigData 就一直可达，不会被回收
```



### 优化：分代回收（Generational GC）

V8 引擎将堆内存分为**新生代**（存活时间短）和**老生代**（存活时间长）。

- 新生代使用 Scavenge 算法，复制存活对象，回收速度快。
- 老生代使用标记-清除和标记-整理，频率更低。

### 如何主动协助垃圾回收？

虽然无法强制立即 GC（`global.gc()` 在 Node 中需特殊标志），但可以：

- 将不再使用的全局变量设为 `null`。
- 解除 DOM 事件监听、定时器、闭包中不需要的引用。
- 使用 `WeakMap` / `WeakSet`（弱引用，不会阻止对象被回收）。

```javascript
// 弱引用示例
let wm = new WeakMap();
let obj = {};
wm.set(obj, "secret");
obj = null;   // obj 被回收，WeakMap 中的条目自动消失，不会阻止回收
```

### 一句话总结

> **垃圾回收是 JS 引擎自动回收“不再可达”内存的机制，主要采用标记-清除算法，开发者无需手动释放内存，但需注意解除无用引用（尤其是全局变量和闭包）来避免内存泄漏。**



## 闭包

**闭包（Closure）** 是 JavaScript 中一个非常重要且独特的特性。简单来说，**闭包就是一个函数，加上该函数定义时所在的作用域中的变量**。更直白地说：**闭包让一个函数能够“记住”并访问它诞生时的外部变量，即使这个外部函数已经执行完毕**。

### 核心机制

在 JS 中，函数可以嵌套定义。当内部函数引用了外部函数的变量，并且这个内部函数被返回或以其他方式“存活”下来时，就形成了闭包。外部函数的执行上下文虽然已经弹出调用栈，但它的变量对象仍然被内部函数引用，因此无法被垃圾回收，继续驻留在内存中。

### 简单示例

```javascript
function outer() {
  let count = 0;  // 外部函数的局部变量
  
  function inner() {
    count++;      // 内部函数引用了外部变量
    console.log(count);
  }
  
  return inner;   // 返回内部函数（闭包）
}

const counter = outer();  // outer 执行完毕，按理说 count 应该销毁
counter();  // 输出 1 —— 闭包“记住”了 count
counter();  // 输出 2 —— 同一个 count 被持续修改
```



### 闭包的常见用途

1. **数据私有化**：创建只能通过特定函数访问的私有变量。

   ```javascript
   function createSecret(secret) {
     return {
       getSecret: () => secret,
       setSecret: (newSecret) => { secret = newSecret; }
     };
   }
   const box = createSecret("123");
   console.log(box.getSecret()); // 123
   box.setSecret("456");
   console.log(box.getSecret()); // 456
   // 外部无法直接访问 secret 变量
   ```

   

2. **函数工厂**：根据参数动态生成功能相似的函数。

   ```javascript
   function multiplyBy(factor) {
     return function(number) {
       return number * factor;
     };
   }
   const double = multiplyBy(2);
   const triple = multiplyBy(3);
   console.log(double(5));  // 10
   console.log(triple(5));  // 15
   ```

   

3. **异步操作中的状态保持**（如定时器、事件监听）：

   ```javascript
   for (var i = 0; i < 3; i++) {
     setTimeout(function() {
       console.log(i);   // 输出三次 3，因为 var 没有块级作用域
     }, 100);
   }
   // 用闭包解决：为每次循环创建独立作用域
   for (var i = 0; i < 3; i++) {
     (function(j) {
       setTimeout(() => console.log(j), 100);
     })(i);
   }
   // 或者直接用 let（块级作用域）
   ```

   

4. **模块化模式**（早期的模块封装）：

   ```javascript
   const calculator = (function() {
     let result = 0;
     return {
       add: (x) => { result += x; return this; },
       subtract: (x) => { result -= x; return this; },
       getResult: () => result
     };
   })();
   ```

   

### 闭包与内存

因为闭包会使外部函数的变量一直存在内存中，如果不当使用（如创建大量闭包并长期持有）可能造成**内存泄漏**（正如你之前问到的）。但合理使用时，这正是闭包的价值所在。

**注意**：在较新的 JS 引擎中，闭包只会保留实际被引用的外部变量，而不是整个父函数的变量对象（称为“闭包优化”）。

### 一句话总结

> **闭包 = 函数 + 对该函数定义时所在作用域（包含外部变量）的引用。它允许函数“记住”定义时的环境，常用于封装私有变量、函数工厂和异步状态保存。但需小心滥用导致的内存占用。**



## 内存泄漏

**内存泄漏（Memory Leak）** 是指程序在运行过程中，**不再需要的内存没有被正确释放**，导致这部分内存无法被操作系统或程序重新利用。随着时间推移，泄漏的内存会越积越多，最终可能造成程序响应变慢、卡顿甚至崩溃。

在 JavaScript 中，由于垃圾回收（GC）是自动进行的，内存泄漏通常是由于**无意间持续持有不再需要的对象引用**，导致垃圾回收器无法判断这些对象是“不可达”的。

### 常见的内存泄漏场景（附 JS 示例）

#### 1. 意外的全局变量

```javascript
function leak() {
  // 没有使用 var/let/const，创建了一个隐式全局变量
  leakedVar = "This is a leak";
}
leak();
// 函数执行完后，leakedVar 仍然挂在全局对象（window/global）上，永远不会被回收
```

**解决方案**：使用 `'use strict'` 避免隐式全局变量，或手动 `delete` / 赋 `null`。

#### 2. 未清理的定时器（setInterval / setTimeout）

```javascript
let largeData = new Array(1000000).fill("*");
setInterval(() => {
  console.log(largeData.length);
}, 1000);
// 定时器回调一直引用着 largeData，即使你不再需要这个定时器，它也会持续运行
```

**解决方案**：不再需要时调用 `clearInterval` / `clearTimeout`。

#### 3. 未移除的事件监听器

```javascript
function bindEvent() {
  let hugeObj = { data: new Array(1000000).fill("*") };
  document.getElementById("btn").addEventListener("click", () => {
    console.log(hugeObj.data.length);
  });
}
bindEvent();
// 即使页面上的按钮被移除，事件监听器仍可能挂在 DOM 节点上（取决于浏览器实现）
// 更常见的是：监听器绑在长期存在的对象上，导致 hugeObj 一直被引用。
```

**解决方案**：使用 `removeEventListener`，或在现代框架中使用“弱引用”方式（如 AbortController）。

#### 4. 闭包滥用

```javascript
function outer() {
  let bigArray = new Array(1000000).fill("*");
  return function inner() {
    // 哪怕 inner 只返回一个简单的数字，但因为内部函数引用了 bigArray，
    // 整个闭包作用域链会一直保留 bigArray
    return 42;
  };
}
let closure = outer(); // outer 执行后，bigArray 本应被回收，但闭包持有了它
```

**注意**：现代 JS 引擎会对闭包做优化（逃逸分析），如果内部函数确实没有使用外部变量，可能不会泄漏。但为了安全，应避免在闭包中挂载大对象。

#### 5. 脱离 DOM 的引用

```javascript
let elements = [];
let btn = document.getElementById("myButton");
elements.push(btn);   // 数组引用了 DOM 节点

// 后来从页面中移除了这个按钮
btn.parentNode.removeChild(btn);

// 但 elements 数组仍然保存着对该 DOM 节点的引用，导致节点无法被垃圾回收
```

**解决方案**：移除 DOM 节点时，同步清除所有对该节点的引用（如从数组中删除）。

#### 6. Map / Set 中过期键

```javascript
let cache = new Map();
function addToCache(obj) {
  cache.set(obj, Date.now());
}
let user = { name: "Alice" };
addToCache(user);
user = null;  // 希望回收 user，但 cache 还持有对原对象的引用，无法回收
```

**解决方案**：使用 `WeakMap` / `WeakSet`（键为弱引用，不影响垃圾回收）。

### 如何检测内存泄漏？

- **Chrome DevTools**：Memory 面板 → 拍堆快照（Heap Snapshot），对比两个时间点的内存变化，找出持续增长且未被释放的对象。
- **Performance 面板**：记录内存时间线，观察内存是否持续上升而不回落。

### 一句话总结

> **内存泄漏是程序中“不再需要但仍被引用”的数据无法被垃圾回收，导致内存占用持续增长。在 JS 中，常见原因包括意外的全局变量、未清理的定时器/事件监听器、闭包滥用、以及 Map 中过期的键值对。避免办法是及时切断无用引用，使用弱引用容器（WeakMap/WeakSet）。**
